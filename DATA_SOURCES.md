> 最後更新：2026-07-09
> 這份是資料蒐集方向的設計筆記，對應 `agent/collectors/` 現有五類 + 延伸構想。
> 目的：把「要抓什麼、去哪抓」和「抓來的東西在報告裡怎麼被使用」分開講清楚，
> 尤其是比較分析題型（雙幣種）的資料設計，也是簡報「數據資料應用」那頁的底稿。
> 六類各自該抓多長的時間窗口（不是每類都套「過去兩週」），見

## 五大方向（現有 collector 對應）

題目本身就講明了要整合價格/鏈上/新聞/社群/總經五類。

### 1. Price 價格

| 需要什麼 | 去哪抓 | 備援 |
|---|---|---|
| 近兩週 OHLCV（漲跌%、最高最低、成交量） | 主辦方 CSV `data/{幣種}_daily_ohlcv.csv`（本地，零網路風險） | — |
| 技術指標（SMA7/14、RSI14、波動率、量能趨勢） | 同一份 CSV 本地純 Python 計算 | — |
| 即時報價（現價、24h 漲跌、24h 量） | CoinGecko `/simple/price` | CryptoCompare `/pricemultifull` |

**這個方向的內容怎麼用：**
- 共同尺度（可跨幣比較）：24h 成交量、量能趨勢、波動率、RSI——同樣公式算出來，天然可比
- 幣種特有：SMA 站上/跌破的意義要搭配該幣種自己的價格區間解讀，不可直接互比數值

### 2. Onchain 鏈上（依鏈路由到不同端點）

| 幣種 | 抓什麼 | 去哪抓 |
|---|---|---|
| BTC | 區塊數、mempool 筆數、24h 交易筆數、算力 | Blockchair `/bitcoin/stats` |
| ETH | 區塊高度、Gas 價格 | publicnode EVM RPC（備援 cloudflare-eth） |
| BNB | 同 ETH | bsc-dataseed（備援 bsc-rpc） |
| SOL | 近期平均 TPS | Solana 官方 RPC |
| XRP | 帳本高度、負載係數、base fee | xrplcluster（備援 s1.ripple.com） |

**這個方向的內容怎麼用：**
- 幣種特有：這整個方向本質上就是「幣種特有層」——不同鏈用不同指標，本來就不可直接互比
- 共同尺度：**目前沒有**——這是最大缺口，見下方待補項目

### 3. News 新聞

2026-07-20 起改為**每幣官方發布源優先**，不再用 CoinDesk/Cointelegraph 全站 RSS + 別名過濾的作法（該作法已淘汰，見下方「已解決」備註）。

| 幣 | 去哪抓 | 型態 |
|---|---|---|
| BTC | Bitcoin Optech Newsletter（`bitcoinops.org/feed.xml`） | RSS，已實測可用 |
| ETH | Ethereum Foundation Blog（`blog.ethereum.org/feed.xml`） | RSS，已實測可用 |
| SOL | Solana Foundation News（`solana.com/news/rss.xml`） | RSS，已實測可用 |
| BNB | BNB Chain 官方 Blog（`bnbchain.org/en/blog`） | 無 RSS，退階解析網頁；標題由網址 slug 還原（非原文） |
| XRP | Ripple Insights（`ripple.com/insights/`） | 無 RSS，退階解析網頁，可取得真實標題與發布時間 |

**這個方向的內容怎麼用：**
- 幣種特有：官方源本身就是「該幣的第一手公告」——升級、監管進度、生態合作、銷毀/供給事件等，直接是層2 特有敘事的原始素材
- 共同尺度：目前沒有——每幣來源網站不同，沒有像過去「同一 RSS、同一過濾規則」下可比的「命中則數」；如需要共同尺度，要另外設計（例如統一改成「本週官方公告則數」這種跨站可比計數）

**✅ 已解決（曾是待修 bug）**：舊版 `news.py` 的 `_matches_coin()` 用短別名（"eth"、"sol"）子字串比對全站 RSS，會誤中一般英文單字。改版後每幣直接讀自己的官方源，不再需要跨幣別名過濾，這個 bug 連同它綁的程式碼一起被移除，不是修好，是問題本身不存在了。

**⚠️ 待驗證**：BNB／XRP 兩個 HTML 解析端點沒有官方保證的穩定 API 合約，站方改版會直接讓抓取失效，賽前建議重跑一次 `scripts/test_collectors.py --coin BNB` 之類確認還能抓到東西。

### 4. Social 社群

| 需要什麼 | 去哪抓 |
|---|---|
| 討論熱度與情緒（標題、score、留言數），近一週最新 5 篇 | Reddit `r/{subreddit}/search.json` |

**這個方向的內容怎麼用：**
- 共同尺度：同樣查詢條件下的 score / 留言數可比（市場關注度代理指標）
- 已知風險：雲端機房 IP 常被 Reddit 擋 403，需標記缺失並在報告揭露

### 5. Macro 總經

| 需要什麼 | 去哪抓 |
|---|---|
| 加密市場恐懼貪婪指數 | alternative.me `/fng/` |
| 美元強弱（對 EUR/JPY/GBP 匯率代理） | Frankfurter |
| （選用）10 年美債殖利率 | FRED（需免費 key） |

**這個方向的內容怎麼用：**
- 共用宏觀背景：這整個方向本質上是「兩幣共用同一份環境描述」，不分幣種——比較題型下**應該只抓一次**，不是每個幣種各抓一次

**⚠️ 待修**：目前 orchestrator 是「每個 collector × 每個幣種」平行跑，macro 不分幣種卻被跑兩次，比較題型會產生重複證據（一模一樣的 Fear & Greed / 匯率各存一筆）。建議比較題型下 macro 只抓一次、兩幣共用。

## 三層用途框架（比較分析題型專用）

上面五大類的資料，在「比較 A 與 B」這種題型下，實際被拿來用的方式分三層：

1. **共同尺度資料**——同一把尺，直接支撐「A 比 B 高/低」這種比較句
2. **幣種特有資料**——各自的尺，支撐「A 的風險特徵是……」這種敘事句，不強求可比
3. **共用宏觀背景**——同一片天空，抓一次兩幣共用，支撐「在當前環境下，誰受益/受害更多」

現有 collector 架構天生對應：price/news/social 主要餵層 1，onchain 主要餵層 2，macro 主要餵層 3。

**現狀最大缺口**：雙幣相對指標（相關係數、beta、最大回撤——兩份 CSV 本地純 Python 算，零 API 成本）目前完全沒有，是唯一能直接量化「風險敞口差異」的共同尺度證據，也是評分「洞察角度」的加分點，建議優先補。

## 新想法：第 6 個方向——Derivatives 衍生品持倉

> 核心洞察：現有五類全是現貨與網路層數據，完全沒有「倉位結構」視角。衍生品數據
> 回答的是「市場上的錢押在哪一邊、擠不擠、誰會被清算」——命中評分「評審未必預期
> 到的切入點」，也是團隊裡 Ken 的 S&T 專業差異化所在。

### 可自建的指標（全部免 key，抓原始數據自己推導，符合紅線）

| 指標 | 邏輯 | 去哪抓 |
|---|---|---|
| 費率擁擠度 | 當前資金費率在近 30 天的百分位。持續高正費率＝多頭擁擠＝下方堆滿清算燃料，利於空頭往下砸（注意：正費率本身是常態，「極端百分位」才是訊號） | Binance `fapi/v1/fundingRate`（有歷史）；備援 Bybit / OKX |
| OI × 價格四象限 | OI↑價↑＝新多進場、OI↑價↓＝新空進場、OI↓價↓＝多頭被清算、OI↓價↑＝空頭回補——四種市場結構各自解讀 | Binance `fapi/v1/openInterest` ＋ 價格 |
| 多空帳戶比／大戶持倉比 | 散戶與大戶方向背離時是強訊號 | Binance `futures/data/globalLongShortAccountRatio`、`topLongShortPositionRatio` |
| 永續基差 | perp 相對現貨指數的溢價/折價 | Binance `fapi/v1/premiumIndex` |
| 跨幣費率差 | 比較題殺手鐧：「A 費率 90 百分位 vs B 40 百分位 → A 倉位擁擠得多、風險敞口更大」——直接量化風險敞口差異的層 1 共同尺度證據 | 同上，兩幣各抓一次本地比 |
| 機構賭盤（選擇權） | IV / put-call / 25-delta skew＝專業玩家開出的漲跌機率盤口 | Deribit 公開 API（**限 BTC/ETH**，SOL 部分）——層 2 幣種特有 |

### 風險與前置驗證

1. **地理封鎖（最重要）**：Binance/Bybit/OKX API 皆擋美國 IP。若部署在 App Runner
   us-east-1 大概率打不通（同 Reddit 403 性質）。對策：部署改 ap-northeast-1（東京），
   或賽前從實際部署環境實測、交易所間互為備援。**正式執行前必須驗證**。
2. **覆蓋不全**：Deribit 選擇權只有 BTC/ETH（+部分 SOL）。這項定位為「有就加分、
   沒有就在限制裡揭露」，不當主力。
3. **紅線確認**：抓的是原始市場數據（費率、OI、持倉比），判斷由自建推理鏈產生，
   不屬於「第三方已產出的完整市場判斷」——合規。

### Hyperliquid 補充

全透明 perp DEX，官方 API 免 key（`api.hyperliquid.xyz/info`），一次救回兩個弱點
（清算數據被 Coinglass 付費牆擋住、Binance 地理封鎖無備援）：

- `metaAndAssetCtxs`：全幣種 funding／OI／24h 量／溢價一個 call 全拿——衍生品主力來源候選
- `fundingHistory`：費率歷史——擁擠度百分位的資料源
- `l2Book`：訂單簿深度——**補上 Kraken/Coinbase 不上 BNB 的洞**（HL 有 BNB perp，賽前 curl 驗證幣種清單）
- 公開 fill 資料：清算成交可見——清算「流」的免費入口
- 新指標：**CEX vs DEX 費率差**（Binance vs HL 同幣費率背離）＝槓桿集中在哪個場地

限制與注意：
1. 「清算地圖」（價位×待清算量熱圖）≠「清算流」（剛發生的清算成交）。前者需重建全場倉位，
   15 分鐘內不可行，仍無免費現成源；我們做的是後者＋funding/OI 間接推論清算壓力，報告措辭別越界。
2. HL 是單一 DEX、高槓桿玩家集中，樣本偏差寫進限制欄。
3. HL 前端擋美國用戶；讀取 API 一般不做地理圍欄，但仍需從實際部署環境實測（同 Binance 驗證一起做）。

### 架構落點

加第 6 個 collector（`derivatives`），完全套現有模式：繼承 `BaseCollector`、
`SourceType` enum 加值、orchestrator 的 `SOURCE_TYPES` 加項。它同時餵兩層：
費率/OI/持倉比＝層 1 共同尺度（最強）、選擇權＝層 2 幣種特有（BTC/ETH）。
來源優先序：Hyperliquid（免 key、無擋、全幣種）與 Binance 互為主備，賽前從
部署環境實測後定案。

## 延伸想法

### 指標產生器——三招變出所有衍生指標

1. **絕對值 → 百分位**：數字本身沒意義，相對自己歷史的位置才有（費率擁擠度）
2. **單序列 → 背離**：兩條本該同向的線走岔了就是故事（價格 vs OI、散戶 vs 大戶）
3. **單幣 → 橫斷面**：同一把尺量兩幣，差距本身就是結論（跨幣費率差）

### 既有來源的便宜升級

- Fear & Greed：`limit=1` → `limit=30`，從「今天的值」升級成「近 30 天百分位」（一行改動）
- 新聞：過濾後「命中則數」本身輸出為量化證據（關注度代理），不只是單則清單
- 新聞語調：推理層讓 LLM 給標題做多空中性分類，產出「新聞語調分數」與價格背離判斷

### 想過但淘汰（簡報答辯用）

| 想法 | 淘汰原因 |
|---|---|
| 清算地圖/爆倉數據 | 完整「地圖」乾淨來源只有 Coinglass（付費 key）；**部分救回**：Hyperliquid 公開 fill 可見清算流（見上方 HL 補充） |
| 交易所淨流入流出（鏈上版） | CryptoQuant/Glassnode 全要錢——**已用免費機構流入代理復活**，見上方「華爾街機構流入代理」 |
| Google Trends | 無官方 API，爬蟲版一次性執行風險過高 |
| ETF 資金流 | 無穩定免費 API，只能靠新聞間接引用 |

### 彩蛋級：日曆效應

- **月相效應**：有正規行為金融文獻（Yuan/Zheng/Zhu 2006 "Are investors moonstruck?"，
  48 國股市滿月前後報酬偏低）。月相是純天文計算（朔日+29.53 天週期），零 API 零風險。
  **正確用法不是當訊號，是當可證偽性展示**：用主辦方 5 年 CSV 把日報酬按月相分桶做檢定，
  在報告限制欄寫「連月相效應都檢驗過，無顯著差異，不納入判斷」——同時展示假設驗證
  方法論、雜訊紀律與幽默感，命中「對不確定性的清醒判斷」。
- **週間效應（體面的親戚）**：週末流動性變薄、成交結構與工作日不同（ETF 後 TradFi
  時段影響變大）。同一份 CSV 分桶檢定「週末 vs 工作日波動與量能差異」，拿得上檯面。
- **農民曆**：簡報彩蛋可以，證據鏈禁入。

### 紀律原則

一次正式執行機會＝每個新資料源都是現場出包點。所有新指標定位為 garnish 不是主菜：
核心五類先保穩，按投入產出比逐一加入，每加一個都要過 dry-run＋真實部署環境實測。
優先序：微結構流動性 → CSV 再榨 → 費率擁擠度 → 其餘。
