## 現況：Price 方向，MA20/60/120

### Step 1：讀 CSV

- 來源：`raw_data/price/{COIN}/{COIN}_daily_ohlcv.csv`（主辦方共同基準資料，5 幣
  種各 1826 筆，2021-06-01 ~ 2026-05-31，UTC，USDT 計價）
- 讀法：`csv.DictReader` 整份讀進來，取 `close` 欄位組成收盤價序列
- 程式碼：`pipeline/compute_ma.py` 的 `load_closes()`

### Step 2：算 MA20/60/120

- `MA20` = 近 20 日收盤均、`MA60` = 近 60 日收盤均、`MA120` = 近 120 日收盤均
- 判讀：現價站上該均線＝該區間動能偏多，跌破＝偏空；三線同向＝排列一致，訊號
  較強
- 程式碼：`pipeline/compute_ma.py` 的 `compute_ma()`
- ⚠️ 這裡跟正式 `price.py` 有落差：正式程式碼的 `load_ohlcv_tail()` 只取最後
  30 列（`INDICATOR_WINDOW=30`），算不出 MA120（需要至少 121 列）。這份
  prototype 直接讀整份 CSV 繞過這個限制，但正式併回 `price.py` 前，這個窗口常數
  要先跟 alanchang 講一聲再改。

### Step 3：生成兩個 evidence 區塊

沿用 `price.py` 現有的兩區塊模式，正式跑起來時這兩塊各自是一筆 `EvidenceDraft`：

1. **區塊①近兩週 OHLCV 摘要**——`build_ohlcv_block()`：期間漲跌%、最高/最低、
   總成交量
2. **區塊②MA20/60/120 位置解讀**——`build_ma_block()`：現價相對三條均線站上/
   跌破

### Step 4：完整歷史序列寫回 raw_data/price/{COIN}/

Ken 提醒：資料其實是 2021-06-01 ~ 2026-05-31（約 5 年，1826 天），一開始寫的
單日快照（`ma_20_60_120.json`，只存「最後一天」一個點）不夠用，**已刪除**。
改成整段歷史每一天都算一次 MA20/60/120，存成序列：

- `raw_data/price/{COIN}/ma_20_60_120_series.csv`：每天一列
  `date,close,ma20,ma20_position,ma60,ma60_position,ma120,ma120_position`
- 前面天數不夠湊滿窗口（例如第 1 天算不出 MA120）該欄位留空，不是 0，避免誤讀
- 程式碼：`pipeline/compute_ma.py` 的 `compute_ma_series()` / `write_series_output()`
- 已核對：BTC 序列最後一列跟前面單點算出來的值一致（MA20=76816.29／
  MA60=76044.92／MA120=72613.46）

## 相對強弱比值 A/B 的 90 天位置

對照 `02_資料來源設計筆記.md`「CSV 再榨」清單第一項、`02改_資料網格.html` 的
`relative-strength` spec：這是「比較題」證據，題目指定比較哪兩幣（A/對手幣）才
決定 A、B，不是固定配對。網格上 5 幣各自掛一顆「X/對手幣」chip，A/B 跟 B/A
雖互為倒數，但證據句分別掛在 A、B 各自名下（題目可能問「BTC 相對 ETH」也可能
問「ETH 相對 BTC」），所以 Prototype 把全部 5×4=20 個方向組合都算出來供查，
不是 10 個不分方向的組合。

### Step 5：算比值序列 + 90 天百分位排名

- 比值 = A 收盤 / B 收盤，逐日算
- 90 天位置 = 當日比值在「近 90 天比值序列」中的百分位排名（window 內
  `<= 當日值` 的比例），前 89 天資料不足留空字串，不是 0
- 程式碼：`pipeline/compute_relative_strength.py` 的
  `compute_relative_strength_series()` / `percentile_rank()`
- 輸出：`raw_data/price/relative_strength/{A}_{B}_series.csv`，每天一列
  `date,ratio,percentile_90d`

## 波動壓縮度百分位

對照 `02改_資料網格.html` 的 `vol-compression` spec（狀態：◐ 本地計算／零外部
API）。不新增來源，複用 Price 方向已下載的 CSV。

### Step 7：近 20 天日報酬標準差 → 90 天百分位

- `vol_20d` = 近 20 天日報酬的母體標準差（%）
- `percentile_90d` = 這個 vol_20d 在「近 90 天 vol_20d 序列」中的百分位排名
  （跟相對強弱比值用同一個 `percentile_rank()` 定義：window 內 `<= 當日值`
  的比例）
- 判讀：百分位低＝當前波動比近 90 天大多數時候更「悶」，是盤整假設驗證題的
  直接彈藥；百分位高＝處於歷史相對高波動期，不適合套盤整假設
- 程式碼：`pipeline/compute_volatility_compression.py`
- 輸出：`raw_data/price/{COIN}/volatility_compression_series.csv`，每天一列
  `date,close,daily_return_pct,vol_20d_pct,percentile_90d`

### ⚠️ 方法論標記（待比較，不是待辦修 bug）

目前用的是最簡單的「歷史滾動標準差百分位排名」，假設報酬序列穩態、對極端值
不敏感。**之後有空要加一版 GARCH(1,1)**（或 EWMA、Parkinson/Garman-Klass 這類
用 high/low 算的 range-based 估計量），跟這版在同一天的數字對照，實測看哪個
對「盤整 vs 噴出」的分類比較準，再決定要不要換掉現在這版簡單版本。這不是現在
版本有 bug，是預留的方法論比較項。

### 目前實際跑出來的結果（2026-05-31，近 20 天窗口，5 幣種）

| BTC | 1.28% | 7.8 | 偏悶（盤整彈藥） |
| ETH | 1.47% | 7.8 | 偏悶（盤整彈藥） |
| SOL | 1.72% | 6.7 | 偏悶（盤整彈藥） |

BNB 百分位 100 跟前面 MA/即時報價章節的觀察一致（BNB 近兩週 +10.41%、三線
均線全站上），波動被自己的近期噴出行情推到近 90 天最高。

跑法：`python pipeline/compute_volatility_compression.py`

## Onchain 鏈上

對照 `02改_資料網格.html` 的「2 Onchain 鏈上」色塊，五幣分屬五條不同鏈、各自
一把尺、共同尺度從缺（02 筆記已認定這是最大缺口）。跟正式
`agent/collectors/onchain.py` 邏輯一致（`_fetch_bitcoin()` /`_fetch_evm()` /
`_fetch_solana()` /`_fetch_xrpl()`），這裡是 Ken 自己驗證用的 prototype。

### Step 8：五幣各自打對應端點，抓鏈上快照

- BTC → Blockchair `/bitcoin/stats`：區塊數／mempool 待確認筆數／24h 交易筆數／
  算力(hashrate_24h)，一次 call 四個欄位都拿到
- ETH → EVM RPC（主 `ethereum-rpc.publicnode.com`，備援 `cloudflare-eth.com`）：
  `eth_blockNumber` + `eth_gasPrice`；有設 `ETHERSCAN_API_KEY` 才疊加供給量補充
  證據，沒 key 就標記 skipped，不影響主要資料
- BNB → EVM RPC（主 `bsc-dataseed.binance.org`，備援 `bsc-rpc.publicnode.com`）：
  同 ETH 邏輯，疊加證據換成 BscScan
- SOL → Solana 公開 RPC `getRecentPerformanceSamples`（近 5 個取樣窗口）：
  `numTransactions / samplePeriodSecs` 逐窗口算完取平均，得到近期平均 TPS
- XRP → XRPL `server_info`（主 `xrplcluster.com`，備援 `s1.ripple.com:51234`）：
  帳本高度(`validated_ledger.seq`)／負載係數(`load_factor`)／
  base_fee(`validated_ledger.base_fee_xrp`)
- 程式碼：`pipeline/fetch_onchain.py`
- 輸出：`raw_data/onchain/{COIN}/onchain_snapshot.json`，跟即時報價一樣本質是
  「當下一個時間點」快照，沒有歷史序列版本，每次跑都會覆蓋掉上一次的結果

### Etherscan API key 補充記錄（2026-07-20）

- Ken 申請了一組 Etherscan key，寫進 `.env` 的 `ETHERSCAN_API_KEY`（`.env` 已在
  `.gitignore`，不會進版控）
- 發現 BscScan 訂閱已併入 **Etherscan API V2**：單一 key 用 `chainid` 參數切鏈
  （1=Ethereum、56=BSC…50+ 條鏈），不用另外申請 BscScan key 了，`fetch_onchain.py`
  已改打統一端點 `https://api.etherscan.io/v2/api`
- 實測發現：免費方案目前只涵蓋 Ethereum（chainid=1）可用；BSC（chainid=56）
  回 `"Free API access is not supported for this chain. Please upgrade your
  api plan..."`，所以 BNB 的供給量補充證據即使有 key 還是 skipped，這不是
  bug，是免費方案的涵蓋範圍限制
- ⚠️ 跟正式 `agent/collectors/onchain.py` 的落差：正式程式碼的 `_fetch_evm()`
  還是打舊版 `api.bscscan.com`、分開吃 `ETHERSCAN_API_KEY`／`BSCSCAN_API_KEY`
  兩個環境變數，併回正式程式碼前要先跟 alanchang 講這個 API V2 遷移，一起
  改掉，不然 `BSCSCAN_API_KEY` 這個設定項之後可能整個沒用

跑法：`python pipeline/fetch_onchain.py`

## CME COT 機構淨倉位

對照 `02改_資料網格.html` 的 `cme-cot` spec（狀態原本是 `unverified`——端點記不
準，這次已實測補上）。跟正式 `agent/collectors/` 還沒有對應模組，這裡先是
prototype。

### Step 9：CFTC 週報打 API，抓 Asset Manager／Leveraged Funds 淨倉位

- 資料源：CFTC 公開 Socrata API（免 key），dataset 是「Traders in Financial
  Futures - Futures Only」（resource id `gpe5-46if`）：
  `https://publicreporting.cftc.gov/resource/gpe5-46if.json`
- 週頻，每週五公布，數據落後到當週二，是層2 敘事的背景佐證，不是即時訊號
- 用 `market_and_exchange_names` 精確比對（不是模糊 `like`），避免混進同幣種下
  的 MICRO／NANO 等其他規格合約；四個名稱已逐一打 API 實測核對過，不是憑印象
  猜的：
  - BTC → `BITCOIN - CHICAGO MERCANTILE EXCHANGE`
  - ETH → `ETHER CASH SETTLED - CHICAGO MERCANTILE EXCHANGE`
  - SOL → `SOL - CHICAGO MERCANTILE EXCHANGE`
  - XRP → `XRP - CHICAGO MERCANTILE EXCHANGE`
- 「機構淨倉位」＝ Asset Manager 分類的 `long - short`；同時存 Leveraged Funds
  淨倉位當對照組（前者偏配置型機構，後者偏方向型/動能型基金，兩者背離本身就
  是訊號）
- 抓近 12 週（約一季）trend，不是只存單週快照，才看得出「連續增加淨多單」這種
  02 筆記寫的判讀邏輯
- 程式碼：`pipeline/fetch_cme_cot.py`
- 輸出：`raw_data/derivatives/{COIN}/cme_cot_snapshot.json`（`asset_mgr_net_latest`
  / `asset_mgr_net_change_latest` / `lev_money_net_latest` /
  `trend_last_12_weeks` / `fetched_at`）
- BNB 沒有 CME 期貨，CFTC 週報查不到，直接跳過——缺席本身是層2 風險特徵（跟 02
  改網格的 BNB override 一致）

### 實測結果（跑於本次對話當下，最新報告日 2026-07-14）

| 幣 | Asset Manager 淨倉位 | 本週變化 | Leveraged Funds 淨倉位 |
|---|---|---|---|
| BTC | +2,815 | +430 | -7,491 |
| ETH | -1,406 | +703 | -7,961 |
| SOL | -263 | +272 | -3,129 |
| XRP | +67 | +187 | -1,921 |
| BNB | — 無 CME 期貨，跳過 | — | — |

觀察：四幣本週 Asset Manager 淨倉位都是往多方變化（`change_latest` 全正），但
絕對淨倉位只有 BTC／XRP 是正的，ETH／SOL 資產管理機構目前仍淨空；Leveraged
Funds 四幣全部淨空且幅度都比 Asset Manager 大，兩個機構分類方向不完全一致，
併回正式 collector 時這個背離本身可以當一條獨立的敘事線。

跑法：`python pipeline/fetch_cme_cot.py`
