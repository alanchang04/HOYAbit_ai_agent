## 1. 解題方向

### 命題核心

命題要求的不是「新聞摘要」或「技術指標回測」，而是：

> 多源整合 → 有層次推理（事實→推論→結論）→ 對不確定性的清醒判斷 → 超越表面資訊的分析角度

### 我們的解法：一套會「自我辯論」的分析 Agent

核心想法很簡單：**單一模型自問自答式的「正反方分析」，本質上是同一套偏好在自己套招**。人類分析師會犯的錯——選擇性引用對自己結論有利的證據、輕描淡寫矛盾訊號——AI 模型一樣會犯。

所以我們把推理鏈拆成有對抗性的角色：

```
事實層（客觀摘要，不下判斷）
  → 交叉驗證層（找出跨來源的一致與矛盾訊號）
    → 推論層：正方分析師 提出最有利論證
              反方分析師 看到正方論證後，具體批評其漏洞、建構反向論證
    → 結論層：綜合辯論結果，給出信心等級、已知限制、可能推翻結論的條件
```

反方不是「隨便寫個反面意見交差」，而是**真的會點名批評正方的具體證據引用**（實測範例見第 5 節）。這是我們認為最能回應評分標準「具洞察的分析角度」與「正反方分析與矛盾訊號處理」的設計。

### 誠實面對不確定性

- 每個結論都標信心等級（高/中/低），且信心等級是**依資料完整度、來源獨立性、矛盾訊號多寡動態校準**，不是模型隨便寫的
- 明確列出「已知限制」與「可能推翻結論的具體條件」——不是空話，而是可觀察、可驗證的條件（例：「若 BTC 恐懼貪婪指數回升至 50 以上」）
- 單一資料來源掛掉（例如 Reddit 被雲端 IP 封鎖）不會被隱藏，會誠實寫進報告的限制欄位

## 2. AI 技術應用

### 為什麼自己刻 orchestrator，不用 LangGraph / Strands

15 分鐘硬性時限、一次執行機會，我們需要對「哪個環節該直接跳過」「幾分鐘後要降級」有精確控制。框架的抽象層在這種場景下反而增加除錯與相依風險。自刻的 asyncio pipeline 讓每個失敗點都可預期、可測試。

### LLM 呼叫架構

```
LLMClient 介面（agent/reasoning/llm_client.py）
    ├── BedrockClient   ← 競賽正式規定唯一合法後端，含 exponential backoff retry
    └── GeminiClient    ← 開發階段暫代測試 prompt 用（正式執行前會切回 Bedrock）
```

兩個 backend 實作同一個窄介面，`pipeline.py` 完全不需要知道底層是哪個模型——這代表評審若要求「換一個 Bedrock 模型試試」，我們只需要改一個環境變數。

### 四步推理鏈的技術細節

- 每一步都要求模型輸出結構化 JSON，並用正則容錯解析（處理模型仍包 code fence 的情況）
- **evidence id 幻覺防護**：模型回應中引用的 evidence id 若不存在於真實證據清單，會被自動過濾掉，而不是讓整份報告因為一個幻覺引用而報錯中斷
- **多層失敗容錯**：單一步驟 JSON 解析失敗 → 重試一次並要求修正格式；辯論步驟（正方/反方）任一方失敗 → 退回單模型一次產出多假設的 fallback；整條推理鏈失敗 → 退化為誠實揭露失敗原因的報告，絕不讓程式崩潰、絕不空手而歸
- **題型自動分流**：依關鍵字判斷多源整合／假設驗證／比較分析，比較分析題型會自動從題目文字偵測第二個幣種（例如「比較 BTC 與 ETH」自動抓出 ETH），並將辯論框架動態改寫成「正方挺 BTC、反方挺 ETH」的具體對抗，而非空泛的正反論證

## 3. 數據資料應用

### 五大類資料來源

| 類型 | 來源 | 特色 |
|---|---|---|
| 價格 | 主辦方共同基準 OHLCV CSV ＋ CoinGecko 即時報價 | **零成本技術指標**：純 Python 計算 SMA7/14、RSI14、日報酬波動率、量能趨勢，決定性運算不受網路/API 影響，穩定性最高的一類證據 |
| 鏈上 | 依鏈路由：BTC→Blockchair／ETH,BNB→EVM 公開 RPC／SOL→Solana RPC／XRP→XRPL RPC | 5 種幣種各自對應正確的鏈上資料來源，非用同一套邏輯硬套 |
| 新聞 | 每幣官方發布源（各幣公司/基金會/鏈官方源，優先 RSS，無 RSS 則解析官方頁面） | 第一手官方公告，不靠關鍵字過濾全站新聞去猜相關性 |
| 社群 | Reddit 公開 `.json` 端點 | 已知限制：部分雲端 IP 會被 Reddit 封鎖，我們誠實揭露而非隱藏 |
| 總經 | Fear & Greed Index ＋ Frankfurter 匯率（美元強弱代理） | 提供宏觀背景，交叉驗證時常被用來解釋市場情緒 |

### 證據可回溯性（Evidence List）

每筆證據都有結構化 schema：`id`／`coin`／`source`／`source_url`／`fetched_at`／`content_reference`／`related_claim`／`source_type`。報告中每個關鍵依據都標註對應的 evidence id，且**組裝報告時會強制檢查**引用的 id 是否真實存在，不存在就視為錯誤——這是命題文件特別強調會抽查的項目。

### 穩定性設計

- 每類資料都有主要來源 + 免 key 備援（例：CoinGecko 失敗退 CryptoCompare）
- 單一 collector 有獨立 timeout（75 秒）與例外隔離，失敗只影響自己那一類證據，不拖垮其他 4 類
- 15 分鐘硬性 deadline，超過 12 分鐘門檻會自動跳過剩餘蒐集、直接進入推理（degraded mode），已用單元測試驗證觸發邏輯正確

## 4. AWS 架構圖

```
                    使用者瀏覽器
                         │ HTTPS
                         ▼
              ┌─────────────────────┐
              │   AWS App Runner     │  ← 跑 Docker image（Web UI + CLI 共用同一份程式碼）
              │  （自動 HTTPS／擴展） │
              └──────────┬───────────┘
                         │
        ┌────────────────┼────────────────────┐
        │ InvokeModel     │                    │ HTTPS（公開 API）
        ▼                 │                    ▼
┌───────────────┐         │         CoinGecko／Blockchair／RSS／
│  AWS Bedrock   │         │         Reddit／Fear&Greed／Frankfurter
│ （Claude 模型） │         │         （5 類資料來源）
└───────────────┘         │
        ▲                  │
        │ instance role    │
        │（bedrock:InvokeModel，不落地任何 AWS key）
        └──────────────────┘

              ┌─────────────────────┐
   本機/CI ──▶│   Amazon ECR         │──▶ App Runner 從這裡拉 image
   docker push│  （Docker image 倉庫）│
              └─────────────────────┘
```

**設計理由**：
- **App Runner 而非 EC2**：不用管理 VM、免自架反向代理與憑證，`docker push` 後幾分鐘就有公開 HTTPS 網址，符合比賽時間有限的情境
- **IAM instance role 而非明碼 AWS key**：App Runner service 掛一個僅有 `bedrock:InvokeModel` 權限的 role，SDK 自動用該 role 認證，現場不用管理任何金鑰外洩風險
- **Web UI 與 CLI 共用同一個 `run_pipeline()`**：不是兩套邏輯，Demo 網址背後跑的就是比賽現場會用的同一套程式碼

## 5. Demo 素材：實測結果摘錄

> 以下為開發階段用 Gemini（暫代 Bedrock）跑出的真實結果，可作為簡報截圖或口頭引用素材。正式 Bedrock 驗證待存取權限確認後補上。

### 範例：BTC vs ETH 比較分析（29 筆真實證據，38.8 秒完成）

**反方具體批評正方的證據解讀**（不是各說各話）：
> 「正方過度解讀了 BTC 近 7 日量能增加 10.13% 的數據，並將其歸因於『更強的市場參與度和流動性』。然而，在價格下跌的背景下量能增加，也可能預示著拋售壓力增大或空頭活躍...」

**最終結論**（有條件的相對判斷，不是各打五十大板）：
> 「在市場情緒極度恐懼且流動性收緊的環境下，BTC 的相對穩定性和機構接受度使其在當前更值得優先關注...而 ETH 則在市場情緒轉暖、對創新和應用增長預期提升時，更值得優先關注。」

### 範例：BTC 假設驗證題型（交叉驗證抓到真實矛盾）

> 「儘管市場情緒處於『極度恐懼』且價格表現疲軟，但近 7 日量能較前 7 日增加 10.13%。通常在恐懼情緒下，交易量可能會下降，但此處交易量增加可能暗示有資金在低位吸籌或市場波動加劇，與單純的恐懼情緒略有不符。」

這類「發現數據與敘事不完全吻合」的觀察，正是命題強調的「跨資料來源連結、非顯而易見的市場脈絡」。
